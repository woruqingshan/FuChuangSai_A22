# API contract v0

Version: **v0** (frozen for local and remote skeleton; breaking changes require a new version file).

## Scope

- Describes the JSON shape between **local `edge-backend`** and **remote `orchestrator`** over HTTP.
- Frontend may send a compatible subset; `edge-backend` is responsible for normalization before calling remote.

## Endpoints (orchestrator)

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/health` | Liveness and minimal status. |
| POST | `/chat` | Turn-based user message in, structured reply out. |
| WS | `/ws/chat` | WebSocket turn-based request/response (same payload shape as `/chat`). |

## Request: `POST /chat`

Content-Type: `application/json`

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `session_id` | string | yes | Stable session identifier for server-side context. |
| `turn_id` | integer | yes | Monotonic turn index within the session. |
| `user_text` | string | yes | Canonical text for this turn. For audio turns, local may leave this empty or pass a browser hint, and remote speech-service will resolve the final transcript. |
| `input_type` | string | yes | Primary turn mode, currently `text` or `audio`. |
| `client_ts` | integer | no | Unix epoch seconds from client when the turn was created. |
| `text_source` | string | no | Source of `user_text`, e.g. `typed`, `browser_speech_api`, `faster_whisper`. |
| `audio_base64` | string | no | Optional raw audio payload for audio turns. |
| `audio_format` | string | no | Audio encoding such as `wav`. |
| `audio_duration_ms` | integer | no | Audio duration in milliseconds. |
| `audio_sample_rate_hz` | integer | no | Audio sample rate. |
| `audio_channels` | integer | no | Number of audio channels. |
| `audio_stream_id` | string | no | Stream identifier for chunked audio turns. |
| `audio_stream_event` | string | no | Stream lifecycle event: `append`, `commit`, or `clear`. |
| `audio_stream_sequence_id` | integer | no | Sequence cursor for chunk ordering when the client emits audio chunks. |
| `audio_chunks` | array | no | Optional chunk list for streaming compatibility. Each chunk can contain base64 payload, format, sequence, and duration metadata. |
| `client_asr_text` | string | no | Optional browser-side transcript hint before local ASR normalization. |
| `client_asr_source` | string | no | Source of the browser-side transcript hint. |
| `audio_meta` | object | no | Audio metadata generated by local capture packaging. |
| `video_frames` | array | no | Compact key frames packaged for this turn. The current local implementation uses a rolling local buffer and sends an event-window subset as JPEG key frames rather than a raw video stream. |
| `video_meta` | object | no | Event-window packaging metadata such as sampled frame count, buffered frame count, dimensions, buffer duration, and key-frame strategy. |
| `speech_features` | object | no | Optional speech-side features. In the target architecture these are usually produced by remote `speech-service`. |
| `vision_features` | object | no | Optional structured visual features. In the target architecture these are usually produced by remote `vision-service`. |
| `turn_time_window` | object | no | Canonical turn timing window for future audio/video alignment and WebSocket migration. It can include trigger time, pre-roll, post-roll, and capture strategy. |
| `alignment_mode` | string | no | Expected multimodal alignment strategy, e.g. `text_only`, `audio_only`, `video_text`, `video_audio`. |

### Example

See `chat_request.example.json`.

## WebSocket: `/ws/chat`

- Client sends one JSON object per turn using the same schema as `POST /chat`.
- Server returns one JSON object per turn using the same schema as the HTTP 200 response.
- If validation fails, server returns:

```json
{"detail":"...","status_code":400}
```

## Response: `POST /chat` (200)

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `server_status` | string | yes | e.g. `ok`, `error`. |
| `reply_text` | string | yes | Natural language reply to show in UI. |
| `emotion_style` | string | yes | High-level style hint for rendering. |
| `avatar_action` | object | yes | Structured hints for digital human layer. |
| `avatar_action.facial_expression` | string | yes | Expression token. |
| `avatar_action.head_motion` | string | yes | Head motion token. |

| `response_source` | string | no | Provider or fallback source, e.g. `mock`, `fallback:provider`. |
| `context_summary` | string | no | Short summary of recent session context used for the reply. |
| `reasoning_hint` | string | no | Lightweight debug hint for orchestration behavior. |
| `turn_time_window` | object | no | Echoed input timing window so downstream renderers can align behavior outputs. |
| `alignment_mode` | string | no | Alignment mode consumed for the current turn. |
| `multimodal_result` | object | no | M5 unified multimodal output contract for downstream emotion/avatar pipelines. |
| `multimodal_result.contract_version` | string | no | Contract version for multimodal payload, currently `m5-v1`. |
| `multimodal_result.alignment_mode` | string | no | Effective alignment mode used in this turn. |
| `multimodal_result.modalities` | array | no | Normalized modality list (`text`/`audio`/`video`) with source, summary, tags and confidence. |
| `multimodal_result.dominant_emotion` | string | no | Fused dominant emotion tag from speech + vision evidence when available. |
| `multimodal_result.fusion_summary` | string | no | Compact machine-readable fusion trace for logs and debugging. |
| `multimodal_result.evidence` | object | no | Structured evidence snapshot including canonical text, speech context, vision context, and frame/audio stats. |
| `multimodal_result.evidence.emotion_inference` | object | no | Optional normalized emotion inference output. Can come from a dedicated emotion service or fallback heuristic. |
| `multimodal_result.evidence.emotion_inference.dominant_emotion` | string | no | Dominant emotion inferred for this turn. |
| `multimodal_result.evidence.emotion_inference.emotion_tags` | array | no | Ordered emotion tag candidates after normalization/deduplication. |
| `multimodal_result.evidence.emotion_inference.confidence` | number | no | Confidence in range `[0,1]` when the upstream model provides it. |
| `multimodal_result.evidence.emotion_inference.source` | string | no | Emotion inference source, e.g. `emotion_service` or `speech_vision_heuristic`. |
| `multimodal_result.evidence.emotion_inference.model_ref` | string | no | Optional upstream model identifier. |

## Health: `GET /health`

### Response (200)

| Field | Type | Description |
|-------|------|-------------|
| `status` | string | `ok` when service is up. |

Optional: `server_time` ISO8601 string for debugging.

Optional extensions:

| Field | Type | Description |
|-------|------|-------------|
| `orchestrator_mode` | string | Current orchestration mode, e.g. `llm-adapter-ready`. |
| `llm_provider` | string | Active provider name configured on remote. |
| `llm_model` | string | Active model name configured on remote. |
| `emotion_service_enabled` | boolean | Whether dedicated emotion-service adapter is enabled. |
| `emotion_service_base` | string | Emotion-service base URL when enabled. |

## Notes

- Keep payloads small: avoid sending full dialog history every turn once server-side session store exists.
- Do not embed large debug blobs in responses intended for the UI path.
- Product rule: one turn should contain either text input or audio input as the primary mode, not both at once.
- For streaming compatibility, a client can send `audio_chunks` with `audio_stream_event=append`, then trigger final recognition with `audio_stream_event=commit`.
- Video is treated as an auxiliary modality for the turn, not an independent primary turn mode.
- The current video path is key-frame based and uses a rolling local buffer plus event-window packaging to stay compact enough for SSH transport.
- `turn_time_window.transport_mode` is currently `http_turn`; future streaming migration may use `websocket_stream`.
- `edge-backend` supports `REMOTE_TRANSPORT=auto` to try WebSocket first and fall back to HTTP when unavailable.
- `turn_time_window.capture_strategy` can distinguish the current `event_window_keyframes` path from future clip or WebSocket streaming paths.
- `avatar_output.websocket_endpoint` and `avatar_output.stream_id` are reserved for future WebSocket-based avatar playback.
- M5 adds `multimodal_result` as a stable cross-service contract so future emotion/avatar models can consume aligned multimodal evidence without changing base chat fields.
- Real-time A2F split-work contract for service/renderer collaboration is documented in `shared/contracts/a2f_ab_contract_v1.md` with streaming examples in `shared/contracts/a2f_ab_events.example.jsonl`.
